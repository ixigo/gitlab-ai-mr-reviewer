## ğŸ“ Commit Guidelines

We use **Conventional Commits** for semantic versioning. Every commit MUST follow this format:

### Commit Format

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

### Commit Types

- **feat**: New feature (triggers MINOR version bump)
- **fix**: Bug fix (triggers PATCH version bump)
- **docs**: Documentation only
- **style**: Code style changes (formatting, etc.)
- **refactor**: Code refactoring
- **perf**: Performance improvements
- **test**: Adding tests
- **build**: Build system changes
- **ci**: CI/CD changes
- **chore**: Maintenance tasks

### Breaking Changes

For breaking changes (triggers MAJOR version bump):

```bash
feat!: migrate to TypeScript

BREAKING CHANGE: Source files are now TypeScript instead of JavaScript
```

### Examples

```bash
# Feature
feat(api): add GitLab inline comment support

# Bug fix
fix(parser): resolve markdown parsing issue

# Breaking change
feat!: remove deprecated API endpoints

BREAKING CHANGE: Old API endpoints have been removed
```

### Protected Branches

**NEVER commit directly to:**
- `main`
- `master`
- `prod`
- `production`

Always use feature branches:
```bash
git checkout -b feat/your-feature-name
```

## ğŸ”„ Release Process

We use **semantic-release** for automated versioning and changelog generation.

### How It Works

1. **Commits trigger releases** based on conventional commit types:
   - `fix:` â†’ Patch release (0.0.x)
   - `feat:` â†’ Minor release (0.x.0)
   - `BREAKING CHANGE:` â†’ Major release (x.0.0)

2. **Automatic actions** on release:
   - Version bump in `package.json`
   - `CHANGELOG.md` generation
   - Git tag creation
   - GitLab release creation

### Branches

- **main/master/prod**: Production releases
- **beta**: Pre-release versions (x.x.x-beta.x)
- **alpha**: Alpha versions (x.x.x-alpha.x)

### Manual Release (Testing)

Test the release process without publishing:
```bash
npm run release:dry
```

## ğŸ”€ Pull Request Process

### Before Creating a PR

1. **Ensure you're on a feature branch**
   ```bash
   git branch --show-current
   ```

2. **Update from main**
   ```bash
   git fetch origin
   git rebase origin/main
   ```

3. **Run build and verify**
   ```bash
   npm run build
   ```

4. **Stage all changes**
   ```bash
   git add .
   ```

5. **Commit with conventional format**
   ```bash
   git commit -m "feat: add your feature"
   ```

6. **Push to remote**
   ```bash
   git push origin feat/your-feature-name
   ```

### PR Checklist

- [ ] Follows conventional commit format
- [ ] No commits to protected branches
- [ ] All TypeScript compiles without errors
- [ ] Code is properly typed
- [ ] Updated documentation if needed
- [ ] Breaking changes are documented

### PR Title Format

Use conventional commit format for PR titles:
```
feat: add inline comment functionality
fix: resolve parser timeout issue
docs: update README with new examples
```
